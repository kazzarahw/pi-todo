# pi-todo — Build Spec

> Design/implementation spec. Fixes decisions, file responsibilities, exact
> interfaces, build order, and acceptance criteria. No function bodies — the
> implementing agent writes code against the contracts.

**Goal:** A todo-list extension the agent uses to plan and track multi-step work, rendered in a persistent widget, with a configurable "keep going" nudge.

**Architecture:** Pure state core (full-list replace reducer + render + nudge logic) behind thin Pi glue (tools + widget + settle hook + factory). State persisted via `pi.appendEntry`, reconstructed on `session_start`.

**Tech Stack:** TypeScript/ESM, Bun + `bun test`, `typebox`, `@earendil-works/pi-*` peers.

## Global Constraints

See `pi-shared/HOUSE-STYLE.md`. Extension-specific application:
- **One tool** `todo_write` (snake_case, `<domain>_<verb>`). No `todo_list`: the write result echoes the full list, and a `<pi-todo>` context injection re-surfaces it after a context reset (Decision 2). `TodoStatus` is a closed set → `StringEnum`.
- Emits `todo:updated { todos }` and `todo:task-complete { task }`.
- Enforcement dial applies (see Decision 3). Config command `/pi-todo`. Log `[pi-todo]`.

## Design Decisions

1. **Full-list replace, not per-item CRUD.** The one tool `todo_write({ todos })` replaces the whole list (idempotent, matches the proven Claude-Code TodoWrite model — no id juggling by the agent) and **echoes the resulting list back in its tool result**, so the agent always sees current state right after writing. IDs are assigned/preserved by the extension, not the agent. **No `todo_list`** — reading is covered by the echo plus the injection in Decision 2.
2. **Persistence via `appendEntry`, visibility via injection.** Each write appends a `todo-state` custom entry; on `session_start`, rebuild from the last `todo-state` entry in `ctx.sessionManager.getBranch()`. Custom entries don't enter LLM context, so to keep the agent aware after a context reset, **inject the current list as a `<pi-todo>` block on `session_start` and after `session_compact`** (harness surface — the read path is a hook, not a tool). The widget (`ctx.ui.setWidget`) keeps the *user* informed in parallel.
3. **Enforcement dial** (`mode`, default `notify`):
   - `off` — tools + widget only, no nudge.
   - `notify` — on `agent_settled` with `in_progress`/`pending` items remaining, inject a `<pi-todo>` reminder via `pi.sendMessage` (steer) to keep going.
   - `block` — same nudge but delivered as a `followUp` that re-triggers a turn (soft-forces continuation). *Note:* todos have no natural `tool_call` to hard-`{block:true}`, so `block` here means "auto-continue," documented as the honest limit.
4. **`task-complete` detection** is a diff: emit `todo:task-complete` for each item transitioning to `done` between the previous and new list — this is what pi-git subscribes to for checkpoints.

## File Structure & Responsibilities

| File | Responsibility |
|---|---|
| `src/state.ts` | `TodoInput` type (re-exports pi-shared `TodoItem`/`TodoStatus`); `applyWrite` reducer (assign ids, diff completions) |
| `src/render.ts` | `renderTodos` → widget lines; `formatInjection` → `<pi-todo>` context block |
| `src/nudge.ts` | `pendingReminder` → reminder string or `null` |
| `src/tools.ts` | `buildTodoTool` → `todo_write` (echoes the list in its result) |
| `src/persist.ts` | append/reconstruct `todo-state` entries |
| `index.ts` | factory: tool + widget refresh + `session_start`/`session_compact` restore-and-inject + `agent_settled` nudge + `/pi-todo` |
| `test/*.test.ts` | state reducer, render, nudge (all pure) |

## Interfaces / Contracts

```typescript
// TodoItem/TodoStatus come from pi-shared (SSOT) — { id, content, status }, content
// (not "text") per the canonical param name and the todo:updated payload type.
// TODO_STATUSES (pi-shared) feeds StringEnum in the tool schema.
import type { TodoItem, TodoStatus } from "pi-shared";
interface TodoInput { id?: string; content: string; status: TodoStatus; }

// state.ts (pure)
function applyWrite(prev: TodoItem[], incoming: TodoInput[]):
  { todos: TodoItem[]; newlyCompleted: TodoItem[] };

// render.ts (pure)
function renderTodos(todos: TodoItem[]): string[];      // widget lines: "▣ done  ◐ in progress  ▢ pending"
function formatInjection(todos: TodoItem[]): string;    // <pi-todo> block via pi-shared injectionBlock ("" when empty)

// nudge.ts (pure)
function pendingReminder(todos: TodoItem[]): string | null; // null when nothing pending/in-progress

// tool.ts
interface TodoDeps { getState; setState; emit; persist; }   // ctx.ui read at call time (widget)
function buildTodoTool(deps: TodoDeps): unknown;        // todo_write({todos:[{content,status,id?}]}) → result echoes the list

// persist.ts
function appendState(pi, todos: TodoItem[]): void;      // pi.appendEntry("todo-state", {todos}) — CustomEntry, no LLM context
function restoreState(ctx): TodoItem[];                 // last todo-state entry in getBranch(), or []

// events: todo:updated {todos: TodoItem[]}; todo:task-complete {task: string}  (task = content, per pi-shared)
```

## Build Sequence

- [ ] **1 · Scaffold** — package.json/tsconfig per house-style; stubs. Accept: `bun install && bun test` runs.
- [ ] **2 · `applyWrite` reducer** — Accept: (a) assigns stable ids to new items, preserves existing ids by text/position; (b) returns `newlyCompleted` = items that became `done`; (c) is pure/deterministic.
- [ ] **3 · `renderTodos` + `formatInjection`** — Accept: `renderTodos` returns one line per todo with a status marker, stable ordering; `formatInjection` wraps the list in a `<pi-todo>` block (empty string when there are no todos).
- [ ] **4 · `pendingReminder`** — Accept: returns a message when any item is `pending`/`in_progress`, `null` otherwise.
- [ ] **5 · Tool** — Accept: `todo_write` updates state via `applyWrite`, emits `todo:updated` + a `todo:task-complete` per newly completed, and its result echoes the current list as text.
- [ ] **6 · Persistence + widget + injection** — Accept: writes refresh the widget via `ctx.ui.setWidget("todo", renderTodos(...))`; `session_start` and `session_compact` restore prior state, repaint the widget, and inject `formatInjection(...)` so the agent re-sees the list.
- [ ] **7 · Nudge hook + config** — Accept: `agent_settled` sends the reminder per `mode`; `/pi-todo` views/sets `mode` (persist like pi-consult's config file at `~/.pi/agent/pi-todo.json`).

## Test Strategy

Pure core (`state`, `render`, `nudge`) fully unit-tested. Tools tested with in-memory `getState`/`setState` fakes and a captured `emit`. Widget/hook wiring verified in the manual smoke test.

## Risks / Notes

- Id stability across full-list rewrites (Decision 1) — reconcile by matching on prior text; document that identical-text reorders may re-id.
- `block` mode is soft (auto-continue), not a hard block — see Decision 3.
