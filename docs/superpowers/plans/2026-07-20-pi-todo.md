# pi-todo — Build Spec

> Design/implementation spec. Fixes decisions, file responsibilities, exact
> interfaces, build order, and acceptance criteria. No function bodies — the
> implementing agent writes code against the contracts.

**Goal:** A todo-list extension the agent uses to plan and track multi-step work, rendered in a persistent widget, with a configurable "keep going" nudge.

**Architecture:** Pure state core (full-list replace reducer + render + nudge logic) behind thin Pi glue (tools + widget + settle hook + factory). State persisted via `pi.appendEntry`, reconstructed on `session_start`.

**Tech Stack:** TypeScript/ESM, Bun + `bun test`, `typebox`, `@earendil-works/pi-*` peers.

## Global Constraints

See `pi-shared/HOUSE-STYLE.md`. Extension-specific application:
- Tools `todo_write`, `todo_list` (snake_case, `<domain>_<verb>`). `TodoStatus` is a closed set → `StringEnum`.
- Emits `todo:updated { todos }` and `todo:task-complete { task }`.
- Enforcement dial applies (see Decision 3). Config command `/pi-todo`. Log `[pi-todo]`.

## Design Decisions

1. **Full-list replace, not per-item CRUD.** Primary tool `todo_write({ todos })` replaces the whole list (idempotent, matches the proven Claude-Code TodoWrite model — no id juggling by the agent). `todo_list()` reads current state (useful after compaction). IDs are assigned/preserved by the extension, not the agent.
2. **Persistence via `appendEntry`.** Each write appends a `todo-state` custom entry; on `session_start`, rebuild from the last `todo-state` entry in `ctx.sessionManager.getBranch()`. Custom entries don't enter LLM context, so the widget + `todo_list` are how the agent sees state.
3. **Enforcement dial** (`mode`, default `notify`):
   - `off` — tools + widget only, no nudge.
   - `notify` — on `agent_settled` with `in_progress`/`pending` items remaining, inject a `<pi-todo>` reminder via `pi.sendMessage` (steer) to keep going.
   - `block` — same nudge but delivered as a `followUp` that re-triggers a turn (soft-forces continuation). *Note:* todos have no natural `tool_call` to hard-`{block:true}`, so `block` here means "auto-continue," documented as the honest limit.
4. **`task-complete` detection** is a diff: emit `todo:task-complete` for each item transitioning to `done` between the previous and new list — this is what pi-git subscribes to for checkpoints.

## File Structure & Responsibilities

| File | Responsibility |
|---|---|
| `src/state.ts` | `Todo`/`TodoStatus` types; `applyWrite` reducer (assign ids, diff completions) |
| `src/render.ts` | `renderTodos` → widget lines with status markers |
| `src/nudge.ts` | `pendingReminder` → reminder string or `null` |
| `src/tools.ts` | `buildTodoTools` → `todo_write`, `todo_list` |
| `src/persist.ts` | append/reconstruct `todo-state` entries |
| `index.ts` | factory: tools + widget refresh + `session_start` restore + `agent_settled` nudge + `/pi-todo` |
| `test/*.test.ts` | state reducer, render, nudge (all pure) |

## Interfaces / Contracts

```typescript
type TodoStatus = "pending" | "in_progress" | "done";   // StringEnum in the tool schema
interface Todo { id: string; text: string; status: TodoStatus; }

// state.ts (pure)
function applyWrite(prev: Todo[], incoming: Array<{ id?: string; text: string; status: TodoStatus }>):
  { todos: Todo[]; newlyCompleted: Todo[] };

// render.ts (pure)
function renderTodos(todos: Todo[]): string[];          // e.g. "▣ done  ◐ in progress  ▢ pending"

// nudge.ts (pure)
function pendingReminder(todos: Todo[]): string | null; // null when nothing pending/in-progress

// tools.ts
interface TodoDeps { getState: () => Todo[]; setState: (t: Todo[]) => void; }
function buildTodoTools(deps: TodoDeps): unknown[];     // todo_write({todos:[{text,status,id?}]}), todo_list()

// persist.ts
function appendState(pi, todos: Todo[]): void;          // pi.appendEntry("todo-state", {todos})
function restoreState(ctx): Todo[];                     // last todo-state entry in the branch, or []

// events: todo:updated {todos}; todo:task-complete {task: Todo}
```

## Build Sequence

- [ ] **1 · Scaffold** — package.json/tsconfig per house-style; stubs. Accept: `bun install && bun test` runs.
- [ ] **2 · `applyWrite` reducer** — Accept: (a) assigns stable ids to new items, preserves existing ids by text/position; (b) returns `newlyCompleted` = items that became `done`; (c) is pure/deterministic.
- [ ] **3 · `renderTodos`** — Accept: returns one line per todo with a status marker; stable ordering.
- [ ] **4 · `pendingReminder`** — Accept: returns a message when any item is `pending`/`in_progress`, `null` otherwise.
- [ ] **5 · Tools** — Accept: `todo_write` updates state via `applyWrite`, emits `todo:updated` + a `todo:task-complete` per newly completed; `todo_list` returns current items as text.
- [ ] **6 · Persistence + widget** — Accept: writes refresh the widget via `ctx.ui.setWidget("todo", renderTodos(...))`; `session_start` restores prior state and repaints.
- [ ] **7 · Nudge hook + config** — Accept: `agent_settled` sends the reminder per `mode`; `/pi-todo` views/sets `mode` (persist like pi-consult's config file at `~/.pi/agent/pi-todo.json`).

## Test Strategy

Pure core (`state`, `render`, `nudge`) fully unit-tested. Tools tested with in-memory `getState`/`setState` fakes and a captured `emit`. Widget/hook wiring verified in the manual smoke test.

## Risks / Notes

- Id stability across full-list rewrites (Decision 1) — reconcile by matching on prior text; document that identical-text reorders may re-id.
- `block` mode is soft (auto-continue), not a hard block — see Decision 3.
